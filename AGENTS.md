## Codex作業ログ運用

Codexは、コードを実装するだけでなく、作業履歴を残すことも成果物の一部と考える。

実装を行った場合は、必ず以下の順で成果を提示すること。

1. タスクログ
2. 実装ログ
3. コミットメッセージ案

### 1. タスクログ

この作業で何を目的として進めたのかを記録する。
コード変更ではなく、開発タスクの進捗を記録すること。

```text
タスク:
- 目的:
- 実施内容:
- 完了状況:
- 次に行うこと:
```

### 2. 実装ログ

実際のソースコード変更内容を記録する。
事実のみを書くこと。

```text
実装:
- 変更ファイル:
- 主な変更:
- 動作確認:
- 未確認事項:
```

### 3. コミットメッセージ案

コミット可能な最小単位ごとにコミットメッセージを提案する。
Conventional Commits を基本とする。

```text
<type>: <summary>

<body>
```

利用する type:

- feat
- fix
- docs
- refactor
- chore
- test

### history への記録

将来的に `history/` ディレクトリへ作業ログを保存することを前提とする。
履歴は時系列で追記し、「何を考え、何を実装したか」が後から追える粒度で記録する。
Gitのコミット履歴を置き換えるものではなく、開発の意思決定を補完する記録として扱う。

#### history ディレクトリ構成

```text
history/
  2026/
    2026-07-03-001.md
    2026-07-03-002.md
```

#### ファイル命名規則

```text
history/YYYY/YYYY-MM-DD-NNN.md
```

- `YYYY` は作業年。
- `YYYY-MM-DD` は作業日。
- `NNN` は同日の連番。001から始める。
- 1つの履歴ファイルには、原則として1つの作業単位を記録する。

#### history ファイル形式

````markdown
# YYYY-MM-DD-NNN

## タスク

- 目的:
- 実施内容:
- 完了状況:
- 次に行うこと:

## 実装

- 変更ファイル:
- 主な変更:
- 動作確認:
- 未確認事項:

## 判断・メモ

-

## コミットメッセージ案

```text
<type>: <summary>

<body>
```
````

```

#### history 記録ルール

- タスクログは、作業の目的と進捗を記録する。
- 実装ログは、変更したファイルと実際の変更内容を記録する。
- 判断・メモには、設計判断、保留事項、迷った点を記録する。
- 実行していない確認作業を、実行済みとして書かない。
- コミット後に履歴を修正する場合は、Git履歴と矛盾しないようにする。

### 運用ルール

- タスクログと実装ログを混在させない。
- 実装していない内容を書かない。
- 推測で動作確認済みと書かない。
- 未確認事項は必ず明記する。
- 複数の独立した変更はコミットを分割する提案を行う。
- README・AGENTS・spec に反する設計変更は行わず、必要な場合は実装前に提案する。
- 設計より実装、実装より小さなコミットを優先する。
```
